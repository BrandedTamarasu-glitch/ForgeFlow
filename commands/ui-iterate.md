<objective>
Convert subjective theme/UI tuning into a measurable optimization loop. Each round, an agent proposes N variants of the target theme, applies each to the running app, captures screenshots, scores against a fitness test suite (WCAG contrast, palette proximity, visual diff, a11y violations), and surfaces the top 3 for the next round. After K rounds, produces a markdown report with ranked candidates.

**Forgeflow integration:** Variants are generated by `lumen-implement` (the Forgeflow team's UX/frontend discipline). Scoring uses Playwright + axe-core — the command bootstraps both if missing. The final report goes to `.forgeflow/<project>/ui-iterations/<date>.md` so it's visible to `compass-review` and `lumen-review` next cycle.

**Prerequisite:** Dev server must be runnable. `/ui-iterate` conflicts with `/fleet` — only run one at a time.
</objective>

<context>
$ARGUMENTS:
- `--theme <file>` — required. Path to the theme/palette file (e.g., `apps/backoffice/src/theme/core/palette.ts`)
- `--variants N` — optional, default 5, max 8 per round
- `--rounds K` — optional, default 2, max 3
- `--target-palette <hex-list>` — optional. Comma-separated hex targets (e.g., `"#7BC96F,#239A3B,#196127"` for GitHub green palette). Used for palette-proximity scoring. If omitted, proximity scoring is skipped.
- `--reference <screenshot-path>` — optional. User-provided reference image for visual-diff scoring. If omitted, the first round captures a reference from the current theme state.
- `--component <selector>` — optional. Narrow the visual-diff capture to a single component (e.g., `"[data-testid='primary-btn']"`). Default: full page.
- `--url <url>` — optional. URL to capture. Default: `http://localhost:5173/` (storefront) or reads from `playwright.config.ts`.
- `--dry-run` — print the plan (variant strategy, scoring rubric, reference-screenshot source) without running iterations.
</context>

## Gotchas
- **Theme file is reverted at the end.** The command doesn't apply the winner — it produces a report and reverts. To apply a winner, copy the palette diff from the #1 section of the report and edit the theme file manually, then commit.
- **HMR must be working.** The 2–5s wait between variants assumes hot module reload picks up theme changes. If HMR is off (or the theme file isn't in the HMR graph), every variant measures the stale baseline. Pre-flight 0a checks dev server health, NOT HMR — verify manually if scores seem uniformly tied.
- **First run bootstraps Playwright + axe-core.** If the project has neither, the command installs `@axe-core/playwright` and writes a spec template. This happens silently on first run — expect a ~30s installation delay on iteration 1.
- **Composite weights are hardcoded (0.35 / 0.25 / 0.15 / 0.25).** If your project cares more about contrast than palette proximity, these bias incorrectly. Edit the weights at Step 2c in the template, or create a project-local override at `.forgeflow/<project>/ui-fitness-weights.json`.
- **Conflicts with /fleet.** Only one process can run dev services at a time. If a `/fleet` run is active, `/ui-iterate` will fail Step 0a's dev server check. Wait for fleet to complete or tear down first.
- **Reference screenshot is captured from CURRENT state on first run.** If the current state is already the baseline you want to beat, that's fine. If you want to iterate toward a design mockup, pass `--reference <path>` with the target image explicitly.

<process>

## Step 0: Pre-flight

### 0a. Verify dev server availability
```bash
# Is the dev server already running on the expected port?
DEV_URL="${URL:-http://localhost:5173/}"
if ! curl -sf -o /dev/null "$DEV_URL"; then
  echo "Dev server not responding at $DEV_URL"
  # Ask user: should we start it?
fi
```
If not running, use `AskUserQuestion`: `"Dev server not detected. Start it now (runs `pnpm dev` in background), or abort?"` If user starts, launch with `run_in_background: true` and wait for health check to pass before continuing.

### 0b. Theme file exists
```bash
[ ! -f "$THEME_FILE" ] && { echo "Theme file not found: $THEME_FILE"; exit 1; }
```

### 0c. Git clean
Working tree must be clean. Theme iteration modifies the theme file many times; any unrelated work-in-progress will get tangled. STOP if dirty:
`"Uncommitted changes detected. Commit or stash before /ui-iterate."`

### 0d. Bootstrap testing infrastructure

**Playwright visual regression:**
Check for `toHaveScreenshot` references in any existing spec. If none:
```bash
mkdir -p tests/ui-fitness/__screenshots__
```
Create a baseline spec file `tests/ui-fitness/baseline.spec.ts`:
```typescript
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

const URL = process.env.UI_ITERATE_URL || 'http://localhost:5173/';
const SELECTOR = process.env.UI_ITERATE_SELECTOR || 'body';

test('visual snapshot', async ({ page }) => {
  await page.goto(URL);
  await page.locator(SELECTOR).screenshot({
    path: `tests/ui-fitness/__screenshots__/current.png`,
  });
});

test('axe accessibility scan', async ({ page }) => {
  await page.goto(URL);
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21aa'])
    .analyze();
  // Write results to JSON for scoring
  await page.evaluate((r) => console.log(JSON.stringify(r)), results);
  // Don't fail the test here — scoring happens separately
});
```

**@axe-core/playwright:**
```bash
if ! grep -q '@axe-core/playwright' package.json; then
  pnpm add -D @axe-core/playwright
fi
```

**Contrast-ratio scoring helper:**
Create `scripts/ui-fitness-score.js` (Node script, no deps) that:
1. Reads a screenshot + extracted palette
2. Computes WCAG AA contrast for each text/bg pair
3. Computes RGB Euclidean distance to `--target-palette` if provided
4. Runs axe violations count
5. Outputs JSON: `{ contrast_pass: n, contrast_fail: m, palette_distance: avg_dist, axe_violations: k, visual_diff_px: p }`

### 0e. Reference screenshot
If `--reference` is provided: validate the file exists, copy to `tests/ui-fitness/__screenshots__/reference.png`.
If not: capture one now from the current theme state using the baseline spec.

## Step 1: Extract current theme tokens

Read `$THEME_FILE` and extract the palette tokens. For MUI `createPaletteChannel`-based palettes, this is the `colors.json` or inline `primary`, `secondary`, `error`, `warning`, `info`, `success` color values.

Store as `tokens_baseline.json` in a temp dir.

If `--dry-run`, print the token snapshot, the planned variant strategy, and exit.

## Step 2: For each round (1..K)

### 2a. Generate N variants

Dispatch `lumen-implement` with this prompt:
```
You are generating theme variants for a fitness-driven UI iteration loop.

Current palette (round {R} baseline):
{tokens_baseline.json verbatim}

Target palette (if provided): {target-palette hex list, or "none"}

Round {R} of {K}. Previous round's top 3 scorers (if R > 1):
{previous round's top-3 variants with their scores}

Generate exactly {N} distinct variants of this palette. Each variant must:
1. Modify color tokens only. Do not touch typography, spacing, or component overrides.
2. Preserve semantic roles: `primary` stays the dominant brand color, `error` stays red-family, `success` stays green-family, etc.
3. Vary meaningfully — if a target palette was provided, move closer to it. Otherwise, explore distinct hue/saturation shifts.
4. Be self-consistent — light and dark mode variants must both be generated and must remain visually harmonious.
5. Maintain WCAG AA contrast between foreground/background pairs (primary text on primary.main, etc.).

Return exactly {N} variants as JSON:
[
  { "name": "variant-1", "palette": { ...full token tree... } },
  { "name": "variant-2", ... },
  ...
]
```

### 2b. For each variant (parallel where safe, sequential if file I/O conflicts)

1. Apply the variant to `$THEME_FILE`:
   - If theme is `colors.json`-driven: overwrite `colors.json` with the variant's palette
   - If theme is inline `createTheme` or `createPaletteChannel` call: use Edit tool to replace the color values
2. Wait for HMR reload (or restart dev server if HMR is off): `sleep 2; curl -sf "$DEV_URL" >/dev/null || sleep 5`
3. Run the fitness suite:
   ```bash
   UI_ITERATE_URL="$DEV_URL" UI_ITERATE_SELECTOR="$SELECTOR" \
     pnpm playwright test tests/ui-fitness/baseline.spec.ts --reporter=json > /tmp/fitness-{variant-name}.json
   ```
4. Score:
   ```bash
   node scripts/ui-fitness-score.js \
     --screenshot tests/ui-fitness/__screenshots__/current.png \
     --reference tests/ui-fitness/__screenshots__/reference.png \
     --palette tokens_{variant-name}.json \
     --target-palette "$TARGET_PALETTE" \
     --axe-results /tmp/axe-{variant-name}.json \
     > /tmp/score-{variant-name}.json
   ```
5. Record: `{ variant: "variant-1", score: {...}, composite: <weighted-sum> }`

### 2c. Composite scoring

Weighted sum, higher is better:
- **Contrast pass rate** (0–1): weight 0.35
- **Palette proximity** (inverted distance, normalized 0–1): weight 0.25 (if target provided; else skip and reweight)
- **Visual diff similarity** (inverted pixel-diff, normalized 0–1): weight 0.15
- **Axe violations** (inverted, normalized 0–1, capped at 20 violations): weight 0.25

Output: sorted variant list by composite score.

### 2d. Select top 3 for next round

Keep the top 3 variants. If this is round K (final), proceed to Step 3. Otherwise, the top 3 become seeds for round R+1 (agent receives them as context for new variants that build on these).

## Step 3: Revert theme to baseline

After the loop completes, restore the theme file to its pre-iteration state:
```bash
git checkout -- "$THEME_FILE"
```
This ensures /ui-iterate leaves no residue. The user picks a winner from the report and applies it manually (or via a follow-up command).
